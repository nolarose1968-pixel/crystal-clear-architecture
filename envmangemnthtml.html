Bun's Automatic .env File Loading

Bun automatically loads environment variables from these files (in order of precedence):

.env
.env.production, .env.development, .env.test (based on NODE_ENV)
.env.local
Note: When NODE_ENV=test, .env.local is not loaded to ensure consistent test environments.

Example .env Files for Fire22 Dashboard

Reading Environment Variables

Bun provides multiple ways to access environment variables:

// Traditional Node.js way
const apiKey = process.env.FIRE22_API_KEY;

// Bun-specific way
const apiKey = Bun.env.FIRE22_API_KEY;

// Also available via import.meta.env
const apiKey = import.meta.env.FIRE22_API_KEY;

TypeScript Support

For TypeScript projects, you can declare your environment variables for type safety: 

// types/env.d.ts
declare namespace NodeJS {
  interface ProcessEnv {
    readonly NODE_ENV: 'development' | 'production' | 'test';
    readonly DATABASE_URL: string;
    readonly FIRE22_API_KEY: string;
    readonly FIRE22_API_SECRET: string;
    readonly JWT_SECRET: string;
    readonly LOG_LEVEL: string;
  }
}

// Alternatively, for Bun.env
declare module "bun" {
  interface Env {
    readonly NODE_ENV: 'development' | 'production' | 'test';
    readonly DATABASE_URL: string;
    readonly FIRE22_API_KEY: string;
    readonly FIRE22_API_SECRET: string;
    readonly JWT_SECRET: string;
    readonly LOG_LEVEL: string;
  }
}

Practical Implementation for Fire22 Dashboard

Here's how you might use environment variables in your dashboard:

// src/config.ts
export const config = {
  env: Bun.env.NODE_ENV || 'development',
  database: {
    url: Bun.env.DATABASE_URL || 'file:./dev.db',
  },
  api: {
    baseUrl: Bun.env.API_BASE_URL || 'http://localhost:3000',
    fire22: {
      key: Bun.env.FIRE22_API_KEY,
      secret: Bun.env.FIRE22_API_SECRET,
    }
  },
  jwt: {
    secret: Bun.env.JWT_SECRET || 'fallback_dev_secret',
  },
  logLevel: Bun.env.LOG_LEVEL || 'info',
};

// Validate required environment variables
if (!config.api.fire22.key || !config.api.fire22.secret) {
  throw new Error('FIRE22_API_KEY and FIRE22_API_SECRET are required');
}

// Usage in your application
import { config } from './config';

const response = await fetch(`${config.api.baseUrl}/agents`, {
  headers: {
    'Authorization': `Bearer ${config.api.fire22.key}`,
  },
});

Bun-Specific Environment Variables

Bun recognizes these special environment variables to configure its behavior:

Variable Description
NODE_TLS_REJECT_UNAUTHORIZED Set to "0" to disable SSL certificate validation (for testing)
BUN_CONFIG_VERBOSE_FETCH Set to "curl" to log fetch request details
BUN_CONFIG_MAX_HTTP_REQUESTS Control maximum concurrent HTTP requests (default: 256)
DO_NOT_TRACK Set to "1" to disable crash reporting and telemetry

‚ö†Ô∏è Important bun pm pkg Limitations

Based on our testing, here are key limitations to be aware of:

# ‚ùå Version Numbers with Dots (Common Issue)
# This will FAIL:
bun pm pkg get metadata.roadmap.v2.2.0        # InvalidPath error

# ‚úÖ Workaround - Access the full object:
bun pm pkg get metadata.roadmap                # Returns complete roadmap

# ‚ùå Array Index Manipulation (Important Limitation)
# Setting array[index] overwrites the entire array:
bun pm pkg set "config.envValidation.required[4]"="NEW_VAR"  # ‚ùå Overwrites array!

# ‚úÖ Best Practice - Set entire arrays:
bun pm pkg set "config.envValidation.required"='["VAR1", "VAR2", "VAR3"]'
üîç Discovered Behavior: bun pm pkg set with array indices doesn't work like JavaScript arrays. It overwrites the entire array instead of extending it.
